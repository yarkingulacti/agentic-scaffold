# Hardening Roadmap — v0.8.1 → v1.0

Audit from 2026-06-18 of gaps identified during a structural review of the
scaffold tool itself. Ordered by priority/impact.

Status legend: ❌ Open · 🔄 In progress · ✅ Resolved

---

## Phase 1 — Safety & Self-Hosting ✅ Resolved in v0.9.0–v0.11.0

### 1.1 `--dry-run` flag ✅ Resolved in v0.10.0

Add `--dry-run` / `-n` boolean option that walks all template directories,
prints the file tree that *would* be written, and exits without touching disk.

- Respects `--only` / `--skip-*` in the preview
- Shows detected profile + file count per component group
- Files: `bin/index.ts`, `src/scaffold.ts`, `src/templates.ts`

### 1.2 Self-host AGENTS.md + CLAUDE.md ✅ Resolved in v0.10.1

The scaffold generates agentic configs for target projects but has none for
itself. This is ironic — the dogfood isn't being eaten.

- Create `AGENTS.md` describing the repo layout, build/test commands, and
  contribution conventions (synthesising existing MEMORY.md + CONTRIBUTING.md)
- Create `CLAUDE.md` as an alias
- Not shipped to npm — `package.json` `"files"` excludes root `.md` files
- Root `AGENTS.md` is hand-written, not generated by the scaffold's own
  templates (no self-recursion)

### 1.3 Unscaffold dirty check ✅ Resolved in v0.11.0

`unscaffold` previously deleted everything without verifying whether the user
had modified files since scaffolding. Now writes a manifest and warns.

- During scaffolding, write `.agentic-scaffold/.manifest.json` with content
  hashes (SHA-256) of every written file + metadata (scaffold version, timestamp)
- During unscaffold, load manifest and compare current files against stored
  hashes
- If a file differs (user modified it), warn and prompt for confirmation
- Files: `src/scaffold.ts`, `src/unscaffold.ts`, `src/fs-utils.ts`,
  `src/config.ts`

### 1.4 Symlink with fallback ✅ Resolved in v0.9.0

Symlinks are fragile on Windows and certain filesystems. The scaffold should
gracefully fall back instead of silently skipping.

- Try `symlinkSync` first, catch `ENOSYS` / `EPERM` / `EACCES`
- On failure, copy the file and log a single warning
- Bonus: detect `process.platform === 'win32'` and skip straight to copy
- Files: `src/fs-utils.ts` → `createSymlinks`

## Phase 2 — Dependency Reduction (next milestone)

### 2.1 Replace Python memory pipeline with Node.js ❌ Open

The scaffold generates Python scripts (`memory_index.py`, `memory_search.py`,
`memory_bundle.py`, `memory_common.py.hbs`) that require Python 3 + optional
`sqlite-vec`. Since the scaffold already requires Node.js, the memory pipeline
should work without a second runtime.

- Rewrite the 4 Python scripts as TypeScript/Node.js Handlebars templates
- Use `node:fs`, `node:crypto`, `node:path` — zero additional npm dependencies
- Replace SQLite dependency with a flat-file keyword index (FTS is overkill for
  agent session memory that's searched by `grep`-equivalent patterns)
- Files: `templates/scripts/*.py` → `templates/scripts/*.ts.hbs`

### 2.2 Opt-in `.history/` and `.scratchpad/` ❌ Open

These directories are created unconditionally during scaffolding. Users should
be able to skip them.

- Add `--skip-history` and `--skip-scratchpad` flags (consistent with existing
  `--skip-*` pattern)
- Include them in the interactive component picker
- Document in README that these directories are created and what they're for
- Files: `bin/index.ts`, `src/scaffold.ts`

## Phase 3 — Polish (v1.0 release candidate)

### 3.1 `NO_COLOR` support + `--json` output ❌ Open

The sci-fi themed ANSI UI is a differentiator but gets in the way of scripting
and accessibility.

- Check `process.env.NO_COLOR` (per no-color.org) and disable all ANSI escapes
  in `src/ui.ts`
- Add `--json` flag: on completion, print `{"written": N, "skipped": M,
  "incompleteFiles": [...]}` to stdout and nothing else to stderr
- Files: `src/ui.ts`, `src/scaffold.ts`, `bin/index.ts`

### 3.2 Skill frontmatter schema validation ❌ Open

The 22 skills are substantive (see e.g. `templates/skills/tdd/SKILL.md` at 109
lines with real workflow guidance) but their frontmatter has no schema. Over
time, diverging conventions will make them harder to maintain.

- Define a JSON Schema for skill frontmatter: `name`, `description`,
  `argument-hint?`, `dependencies?`, `tags?`
- Add `bin/validate-skills.ts` that parses all `templates/skills/*/SKILL.md`
  frontmatter, validates against schema, and reports errors
- Add `validate-skills` npm script and run it in CI (`biome` step) and
  `prepublishOnly`
- Not a runtime change for users — maintainer tooling only
- Files: `bin/validate-skills.ts`, `package.json`, `.github/workflows/publish.yml`

## Explicitly deferred

| Audit finding | Rationale |
|---------------|-----------|
| Handlebars is too simple for 50+ templates | Fine at current scale; revisit if templates grow 3× |
| Remove the sci-fi themed UI | Personality differentiator; `NO_COLOR` + `--json` suffice |
| Skills are placeholders / low quality | Disproven on inspection — skills are genuinely useful (TDD: 109 lines, implement: 17 lines, bugfix: 13 lines); only lacking schema validation |
| Memory pipeline dependency on sqlite-vec | Graceful FTS5 fallback already exists; full Node rewrite in Phase 2 eliminates Python requirement entirely |
